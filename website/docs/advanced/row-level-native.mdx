# PostgreSQL Native Row-Level Security

`RowLevelSecurityNative` pushes tenant isolation **into PostgreSQL**. Each query
runs in a transaction that first calls `set_config('app.tenant_id', <tenant>, true)`,
and a `CREATE POLICY` on each tenant table filters rows against that setting. The
payoff over the client-side strategy: Quark **cannot bypass the policy even from
`client.Raw()`** — the engine itself enforces isolation.

It's PostgreSQL-only. On the other five engines, use
[`RowLevelSecurityClient`](multi-tenant#row-level-isolation) (client-side WHERE
injection); attempting Native elsewhere fails at construction:

```go
_, err := quark.For[Order](tenantCtx, nativeRouter).List() // non-PG base client
// errors.Is(err, quark.ErrUnsupportedFeature) == true
```

## Native vs Client — when to use which

| Concern | `RowLevelSecurityNative` | `RowLevelSecurityClient` |
| --- | --- | --- |
| Where the filter runs | PostgreSQL engine | Quark query builder |
| `client.Raw()` / `Exec()` | Filtered by the policy | Bypasses the predicate |
| Dialects | PostgreSQL only | All six |
| Setup | One `CREATE POLICY` per table | None |
| Per-query cost | Implicit transaction + `set_config` | None |
| Use when | Production; bypass risk matters | Cross-dialect dev / staging |

They're mutually exclusive per router — Native isn't a drop-in upgrade of Client.
The two enforce isolation in different places: Native leans on real PostgreSQL
policies and a transaction-scoped session variable, so the database itself
rejects rows the tenant may not see, even through raw SQL. Client mode only adds
a predicate to queries Quark builds. Switching means installing policies, not
flipping a stricter flag.

## Setup

### 1. Install the policy on each tenant table

Either run the DDL yourself, or embed the `quarktenant` library and let it emit
the DDL from your registered models.

**Option A — `quarktenant` (recommended).** A tiny `main.go` registers your
models and delegates to `quarktenant.Run`, which emits one
`ALTER TABLE … ENABLE/FORCE ROW LEVEL SECURITY` pair plus one
`CREATE POLICY <table>_tenant_isolation` per table:

```go
func main() {
    client, _ := quark.New("pgx", os.Getenv("QUARK_DSN"),
        quark.WithLimits(quark.Limits{AllowRawQueries: true}))
    defer client.Close()

    _ = client.RegisterModel(&models.Order{}, &models.Invoice{})
    os.Exit(quarktenant.Run(context.Background(), os.Args[1:], client))
}
```

```bash
go run ./cmd/tenant install-rls-policies --dry-run   # print the DDL, no change
go run ./cmd/tenant install-rls-policies             # apply (takes a migration lock first)
go run ./cmd/tenant install-rls-policies --tenant-col=org_id --native-rls-var=app.org_id
```

A runnable example is at
[`examples/tenant-rls-native/`](https://github.com/jcsvwinston/quark/blob/main/examples/tenant-rls-native/main.go).

**Option B — manual DDL** through your normal migration pipeline:

```sql
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders FORCE ROW LEVEL SECURITY;

CREATE POLICY orders_tenant_isolation ON orders
    USING (tenant_id = current_setting('app.tenant_id', true)::text)
    WITH CHECK (tenant_id = current_setting('app.tenant_id', true)::text);
```

Two parts matter: `WITH CHECK` stops a tenant from *writing* rows for another
tenant (without it, isolation is read-only), and `FORCE ROW LEVEL SECURITY`
removes PostgreSQL's default exemption for the table owner — which your app role
usually is, so without `FORCE` the policy would be decorative.

:::caution UUID / BIGINT tenant IDs need an explicit cast
`current_setting()` returns TEXT. A TEXT/VARCHAR tenant column works with the
default `::text`; a `UUID` or `BIGINT` column needs `--cast=uuid` (or `::uuid` /
`::bigint` in manual DDL), or the comparison fails and the policy returns zero
rows.
:::

### 2. Configure the router

```go
cfg := quark.DefaultTenantConfig()
cfg.Strategy = quark.RowLevelSecurityNative
cfg.BaseClient = pgClient
// cfg.NativeRLSVar = "app.tenant_id"  // default; override to match your policy
router := quark.NewTenantRouter(cfg, ResolveTenant, nil)
```

### 3. Query — both paths set the tenant transparently

```go
// Recommended for multi-step work: one transaction, one set_config.
err := router.Tx(tenantCtx, func(tx *quark.Tx) error {
    orders, err := quark.ForTx[Order](tenantCtx, tx).Where("status", "=", "paid").List()
    // ...
    return err
})

// Single short read: implicit transaction per call.
orders, _ := quark.For[Order](tenantCtx, router).Where("status", "=", "paid").List()
```

Both reach PostgreSQL with `app.tenant_id` set to the resolved tenant; the policy
filters server-side.

:::note Use `router.Tx` for anything non-trivial
`For[T](ctx, router)` wraps each call in an implicit transaction (needed because
`set_config(..., is_local=true)` only applies inside one). That transaction is
scoped to the operation: it commits an instant after the call completes — the
commit is deferred, not part of the statement itself (see the write-semantics
limitation below). `Iter`/`Cursor` are the exception: their transaction stays
open until you close the cursor, holding a pooled connection and shared locks
on the table for as long as the stream lives. For multi-step work, batch jobs
and streaming, run inside `router.Tx` — one transaction, opened and committed
exactly where you can see it.
:::

## Raw SQL under Native

`client.RawQuery` / `client.Exec` go straight to the pool, so they skip the
implicit transaction that sets `app.tenant_id`. Unlike the client-side strategy,
this is **not a hole** under Native: with no tenant set, the policy's `USING`
returns zero rows and an `INSERT` fails `WITH CHECK` — the engine stays in
control. It's still rarely what you want (you lose the tenant scope and the
builder), so a Native router's base client logs a warning when raw SQL runs with a
resolvable tenant:

```
event=quark.tenant.raw_under_native_rls op=RawQuery tenant=acme
```

For tenant-scoped access, use the builder against the router (`For[T]` or
`router.Tx` + `ForTx[T]`). `client.Raw()` takes no context, so it's never routed
through `set_config` — reserve it for schema/maintenance, not tenant data.

## Troubleshooting

**Query returns zero rows when it shouldn't** — the usual sign Native isn't fully
wired:

1. The policy isn't installed — check `SELECT * FROM pg_policies WHERE tablename = '<table>'`.
2. `set_config` never ran for this query — raw SQL outside `router.Tx` / `For[T]`; the setting reads `NULL` and filters everything out.
3. `FORCE ROW LEVEL SECURITY` is missing and the caller owns the table — re-emit the DDL with `FORCE`.

**A migration or DDL statement hangs** (`ALTER TABLE`, `DROP TABLE`, an index
build) — something is holding a lock on the table, and under Native the usual
suspects are implicit transactions that have not finalized yet:

1. An open `Iter`/`Cursor` stream. Its transaction — connection and
   `ACCESS SHARE` lock included — lives until the cursor closes (an abandoned
   cursor is reclaimed when `QueryTimeout` expires). Close cursors promptly,
   or run the stream inside `router.Tx`.
2. On releases up to 1.3.2, `Create` (and an `Update` falling back to an
   insert) kept its implicit transaction open until the **caller's** context
   ended. With a long-lived context — a CLI tool or a batch job on
   `context.Background()` — every insert parked one `idle in transaction`
   session, and any later DDL on the table blocked behind their locks
   indefinitely. Current releases scope that transaction to the operation, so
   it drains on its own; on affected versions, give each operation a
   short-lived context or write through `router.Tx`.

To see the culprits:
`SELECT pid, state, query FROM pg_stat_activity WHERE state = 'idle in transaction'`
— each row is a transaction holding its locks with no statement running.

**`ErrUnsupportedFeature: RowLevelSecurityNative requires PostgreSQL`** — the
router's base client isn't on PG. Switch to `RowLevelSecurityClient` for non-PG.

**`new row violates row-level security policy`** — the row's `tenant_id` disagrees
with `current_setting('app.tenant_id')`; make sure the field is set and matches
the resolver for the current context.

## Migrating from the client-side strategy

1. Install the policy + `FORCE ROW LEVEL SECURITY`.
2. Switch the router to `RowLevelSecurityNative` on the next deploy.
3. Re-audit `client.Raw()` callers — under Client they bypassed the builder's
   filter; under Native the engine's policy now catches them.
4. Update any test that asserted the literal `WHERE tenant_id = ?` text — the SQL
   no longer carries it (the engine does).

## Limitations

- **No `LISTEN/NOTIFY` integration here** — outbound CRUD events ship via
  [`Client.UseEventBus`](events).
- **Policies aren't auto-removed** when you drop the router; manage them in your
  schema migrations.
- **Streaming holds its transaction open.** `Iter`/`Cursor` under the
  implicit-transaction path keep their transaction — connection and shared
  locks included — until the cursor closes; `QueryTimeout` bounds an abandoned
  one. Every other operation releases its transaction as the call completes.
  For long streams, prefer `router.Tx`.
- **Write semantics under the implicit `For[T]` transaction.** A write that
  goes through the query paths — `Create`, which runs `INSERT … RETURNING`,
  and the multi-row `CreateBatch` — executes inside an implicit transaction
  whose commit is **deferred**: it lands an instant after the operation
  returns, not inside the call itself. Once the statement has executed,
  cancelling the request does **not** roll the write back. This diverges from
  the exec path (`Update` / `Delete`), which commits synchronously before
  returning and is rolled back like any interrupted statement. Two
  consequences to plan for:
  - There is a brief window after the operation returns in which the write is
    not yet visible to any other transaction — including your own next query.
    It closes in milliseconds, but "read your own write on the very next
    line" is only guaranteed inside `router.Tx`.
  - If the deferred commit fails, the operation has already reported success.
    Quark logs the failure at `ERROR` through the client logger and increments
    [`Client.DeferredCommitFailures`](../reference/api/client#deferredcommitfailures)
    — alert on that counter; each unit is a write the engine rolled back after
    the caller saw it succeed.

  `router.Tx` has none of this: it commits synchronously inside the callback.
  Prefer it for any write whose durability must be settled before you answer.
- **A driver panic hands cleanup to a detached goroutine.** If the driver
  panics mid implicit transaction, Quark rolls the transaction back and
  returns its pooled connection on a detached goroutine — after a panic,
  `database/sql` may still hold internal locks that would deadlock a
  same-goroutine rollback. When those locks are never released, that cleanup
  blocks and the connection stays out of the pool. A watchdog makes this
  observable: a cleanup that overruns the client's `QueryTimeout` is logged
  once at `ERROR` and counted in
  [`Client.BlockedPanicCleanups`](../reference/api/client#blockedpaniccleanups)
  — alert on that counter alongside `DeferredCommitFailures`; each unit is a
  held connection you may need to recycle by restarting the process. The
  cleanup keeps retrying either way.
- **Single-row reads distinguish "no rows" from an isolation failure.** When a
  single-row operation (`Count`, aggregates, `Create` reading back
  `INSERT … RETURNING`) cannot even set the tenant variable — acquiring the
  connection, opening the implicit transaction, or the `set_config` call
  itself fails — the operation returns that error, prefixed with the failing
  stage (`native rls: begin tx: …`, `native rls: set_config: …`) and with the
  original cause intact for `errors.Is`. It is never reported as
  `sql.ErrNoRows`, so an empty result always means exactly that: the policy
  filtered every row.
